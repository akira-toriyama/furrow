# furrow — Glossary

> furrow's *ubiquitous language*: the canonical word for each concept, and what
> it precisely means. Use these terms in code, commits, docs, and `--json`
> field names so the same thing is never called two things. Borrowed from a
> field of soil — parallel **furrows** are the status **lanes**, and you work
> the backlog by driving one lane forward at a time.
>
> Field names below match `internal/core/task.go` exactly (the struct field
> order *is* the JSON key order — see `internal/core/marshal.go`).

## Terms

| Term | Meaning |
|---|---|
| **task** | One tracked item — furrow's unit of work. The `core.Task` struct (metadata only): `id`, `title`, `status`, `priority`, `value`, `effort`, `labels`, `repos`, `parent`, `deps`, `refs`, `checklist`, `created`, `updated`, `closed`, `body`, `type`. Its long-form prose is *not* a field — it lives in a separate body file. |
| **type** | The optional `type` field: the work-item TYPE, a value from the closed `[types].order` vocabulary (default `task`, `epic`). It is a **declaration**, not an inference from structure — a task is an epic because it says so, not because it has children. Empty (`omitempty`) reads as the configured default. A typo is rejected (`--type epci` → exit 2 + candidates, like an unknown lane); `furrow lint` warns `unknown-type` on a hand-edited stray. Set it at `add --type` / `set --type`, filter with `ls --type`. |
| **container / epic** | A `type` listed in `[types].containers` (default: `epic`) — a **box** that groups child work rather than being worked itself. `furrow next` skips containers (surface a ready one with `next --containers`); `furrow ls --tree` shows a container's rolled-up child `progress` (`done/total`) and marks it **stuck** when it has open work but no actionable descendant; `furrow revisit` signals `children_done` (all children done — consider closing) and `stuck_container`. A container is closed by a human like any task — furrow never auto-closes a box whose children are all done. |
| **index** | The in-memory aggregate folded from the per-task **shards** — there is no `.furrow/index.json` file. Structured metadata only. Each shard is machine-written through one deterministic path (`core.MarshalTask`); **never hand-edit a shard** (a manual edit churns the git diff and can break the byte-for-byte contract). Query the aggregate with `furrow ls`/`show`/`next --json`. |
| **shard** | The metadata for one task on disk: `.furrow/tasks/<id>.json`, a bare `core.Task` object (no `schema_version` wrapper). One shard per task, 1:1 with its `bodies/<id>.md`. Machine-written by `core.MarshalTask`; **never hand-edit it**. One file per task means distinct ids touch distinct files, so a git merge across worktrees/PRs is a conflict-free union — the win `bodies/<id>.md` already had, now extended to metadata. |
| **meta** | `.furrow/meta.json` — `{"schema_version": 5}`, the single board-wide layout version, in its own file (never inside a shard) so a bump touches one file and no shard becomes a git merge point. Machine-written; **never hand-edit it**. The number is the **board's**, not the binary's, and an **input to every write, never an output**: an older board is readable but **read-only** (`schema-upgrade-required`, exit 2), a newer one is refused (`schema-too-new`, exit 3), and only **`furrow upgrade`** (a **flag day**, below) raises it — an ordinary command stamps it on a fresh store (`furrow init`) and otherwise never touches it. An unreadable `meta.json` is an error, not a guess. The two-sided version gate is documented in [`architecture.md`](architecture.md). |
| **flag day** | The moment a board's **meta** version is raised — the one-way, coordinated cutover `furrow upgrade` performs (never a background migration; furrow never auto-migrates). After it, *no older furrow can write that board*, including anything pinned to an earlier release (a repo's `sync-task-status.yml@vX.Y.Z` CI). furrow cannot see those pins, so the ordering is a human's: **(1)** release the new furrow, **(2)** bump every caller's pin, **(3)** only then `furrow upgrade --yes` + `furrow sync`. `upgrade` previews unless `--yes` and prints exactly that checklist; it is idempotent, and there is **no downgrade** — recovery is `git revert` on the board repo. Meanwhile the board is **readable but read-only**, which `furrow board` reports (`schema_state: outdated`, `writable: false`) and `furrow lint` warns about (`schema-outdated`) without erroring. Rationale in [`non-goals.md`](non-goals.md); mechanism in [`architecture.md`](architecture.md). |
| **unknown-key passthrough** | The rule that a shard's **top-level** keys this binary does not know are **preserved, not dropped**: `core.Unmarshal*` parks them and `core.Marshal*` re-emits them (sorted, after the known keys), so an old binary hands a future field back exactly as it found it. The other half of the version gate: **the gate stops a *bumped* layout from being misread; the passthrough stops an *unbumped* one from being destroyed.** Its limits are part of the term: **top-level only** (a key inside a `checklist` item is still dropped), **not retroactive** (every furrow ≤ `v0.9.0` still destroys them, so a shared board is safe only once every writer — pinned CI included — has it), and **preserving is not honouring** (an old binary carries a future `"blocked": true` and still shows that task in `furrow next`). Hence the bump rule stands, with `TestShardFieldsGolden` as its teeth. It covers all **three** machine-written files (a task shard, a `repos/` review shard, `meta.json`) and `furrow lint` warns **`unknown-shard-key`** on each. The full mechanism — why "known?" must use `strings.EqualFold` (`encoding/json`'s own Unicode case-folding, not `strings.ToLower`), with the U+017F / U+0130 corruption cases — is in [`architecture.md`](architecture.md). |
| **extras** | The unexported carrier the passthrough parks unknown keys in (`core.Extras`, a `map[string]json.RawMessage`; `nil` when there are none — the normal case). It is **not a field on disk**: `encoding/json` cannot see an unexported field, so it can never surface as a literal `"extras"` key nor leak into a `--json` view (which is also why **`Task` must never grow a `MarshalJSON` method** — see [`architecture.md`](architecture.md)). The only way *out* is `ExtraKeys()`, on `Task`, `RepoRecord` **and** `Meta`; there is deliberately no way *in* other than reading a file — furrow can hand an unknown key back, never invent one. |
| **body** | The long-form prose for one task, stored as `.furrow/bodies/<id>.md` and addressed by the task's `body` field as a *relative path* (e.g. `bodies/t-0042.md`), never inline content. Plain markdown, no escaping — **bodies are hand- and agent-editable**. The split (metadata in the shard, prose in the body) is the whole point of the hybrid store. |
| **lane** (== status) | A workflow column. A task's `status` field holds one lane value; the set of lanes and their top-to-bottom order is defined by `[lanes].order` in `config.toml` (default: `inbox`, `backlog`, `ready`, `in-progress`, `waiting`, `done`, `icebox`). "Lane" and "status" are the same thing — the field is `status`, the concept is a lane. |
| **waiting** (lane) | The GTD *Waiting-For* lane: a task delegated or blocked on **someone/something external** (distinct from a `dep`, which is an internal blocking task). It is **terminal** (not in `next`) and parked — it does **not** stamp `closed` and is never archived. Move a task there with `furrow move <id> waiting`; move it back when unblocked. |
| **priority** | A *sparse* integer that orders tasks within a lane (lower sorts higher). New tasks are spaced by `[priority].step` (default 10), starting at `[priority].default` (100): 100, 110, 120… The gaps mean **reorder edits one field** instead of renumbering neighbors. Set it with `furrow reorder <id> <priority>`, or relatively with `furrow reorder <id> --before/--after <ref>` (same lane only): furrow picks the midpoint, and only when the gap is exhausted does it respace the whole lane — atomically in the same write, the neighbors' moves reported in the `--json` envelope's `renumbered` array (their `updated` deliberately does not advance). `furrow set -s <lane> --before <ref>` combines a lane move with the placement (relative to the *destination* lane) in one write — the cross-column drop. |
| **id** | A task's permanent handle, e.g. `t-k3m9p` — the `id` field and the stem of both `tasks/<id>.json` and `bodies/<id>.md`. **Frozen**: assigned once and **never reused or renumbered**. It is a **random** value (`[ids].prefix` + a random Crockford-base32 suffix of `[ids].width` chars), generated locally with no shared counter, so concurrent `furrow add` from separate worktrees can't collide. Legacy zero-padded numeric ids (`t-0042`) stay valid and coexist. Identity is stable across moves, reorders, and renames. |
| **label** | A free-form tag in a task's `labels` array (sorted + deduped on write, so a set is a set). Categorize and filter with `furrow ls -l <label>`. When `[labels].required = true` in config, every task must carry at least one label — `furrow add` rejects a label-less task (exit 2) and `furrow lint` flags one. A repository is **not** a label — that is the `repos` field. Default is not required. |
| **repo** (the `repos` field) | A repository a task relates to, as an `owner/repo` identifier (e.g. `akira-toriyama/furrow`) in the task's `repos` array — a first-class set (sorted + deduped, `[]` never null), 0..N per task. A repo is **not** a label — labels stay pure free-form tags. Attach/detach with `furrow repo <id> --add/--rm` (or `furrow add -r`); filter and scope reads with `-r`, taking a full `owner/repo` or a short name that resolves case-insensitively at a `/` boundary against the board's known repos (ambiguity is exit 2 with a `candidates` array). An active central board/pointer unions its scope repo into `repos` on `add`. An empty set means a **draft**. `furrow lint` warns on an entry that is not owner/repo-shaped. |
| **draft** | A task whose `repos` set is empty — attached to no repo yet, furrow's analogue of a GitHub Issues *draft*. First-class, not an error state: create one with `add --draft` (which suppresses the board's scope-repo union — also what `add` falls back to when a board cannot derive its repo), list them with `ls --drafts` (bypasses the board scope), and `revisit` flags every open one with the `no_repo` signal regardless of scope. Promote a draft by attaching a repo (`furrow repo <id> --add owner/repo`). When a repo scope (explicit `-r` or the board's auto scope) hides drafts on `ls`/`next`, a one-line stderr hint points at `--drafts`. |
| **checklist item** | A `core.ChecklistItem` (`text` + `done` bool) inside a task's `checklist` array — a unit of work small enough to tick off without spawning its own task (furrow's take on a GitHub "Sub-issues progress" line). Append one with `furrow check <id> --add "..."`; toggle by zero-based index with `furrow check <id> <item-index>` (`--off` to uncheck). |
| **ref** | An entry in a task's `refs` array: a pointer *out* of furrow — a `file:line` location (e.g. `docs/x.md#L10`) or a URL. References context; does not affect ordering or `next`. Add with `furrow add --ref ...`. |
| **asset** | A media file attached to a task with `furrow attach <id> <file>`: copied into `.furrow/bodies/assets/` as `<id>-<sanitized-name>` (atomic, collision-free basename — written only via `Store.SaveAsset`) and referenced from the task's **body** by a relative markdown line (`![name](assets/<id>-<name>)` for images, a plain link for other media). It never touches the `refs` array. LFS-independent: a plain file copy plus a body edit — a `.gitattributes` rule can hand the blob to git-lfs transparently. `furrow lint` warns (never errors) on a **dangling** body reference to a missing asset file, an **orphan** asset no body references, and an **oversized** asset above the warning threshold. `attach --json` emits `{id, asset, ref, line}`. |
| **dep** (dependency) | An entry in a task's `deps` array: the **id** of another task this one waits on. A task is only actionable once *every* dep is in the done lane (an unknown or not-yet-done dep blocks it). This is what `furrow next` reasons over. Add with `furrow add --dep ...`. |
| **parent** | The optional `parent` field: the **id** of an enclosing task, for grouping sub-tasks under a larger one. Hierarchy only — it does not gate `next` the way a dep does. Omitted entirely when absent (`omitempty`). Set it at `add --parent`, change or clear it later with **`furrow parent <id> <parent-id>`** / `--rm`, read both directions (parent + children) with `furrow parent <id> --list`, and draw the whole hierarchy with **`furrow ls --tree [<id>]`** (where `parent` is the tree's *skeleton* and `deps` are the *gate* that rides along as each node's `blocked_by`). Re-parenting is acyclic: a cycle has no root, so every task in one would belong to no tree at all (`furrow lint`'s `parent-cycle`, an error, is the backstop for the half-edges a git merge can close behind the app). A **done** parent is allowed — filing a leftover under the epic it came from is a legitimate record — and an open task left under one is `furrow lint`'s `parent-done` warn, the hierarchy twin of the **reconcile gap**. |
| **link** | A `[[<id>]]` reference embedded in a task's **body** prose — a task→task pointer, resolved by `core.LinkPattern`/`core.ExtractLinks` in `internal/core/links.go` (the single source both readers share). Only the explicit brackets count: a **bare** id is deliberately *not* a link (so an agent's id-laden progress log never reads as a mention), and a `[[id]]` written inside `backticks` or a fenced code block is an inert example, not a live link. It powers `show --backlinks` and `furrow lint`'s dangling-link check — which warns (`SevWarn`) on a **dangling** link (a `[[id]]` whose target is neither a hot task nor an archived id) while a link to an **archived** task is fine. Distinct from a **ref** (a pointer *out* of furrow at a `file:line`/URL) and a **dep** (a blocking edge that gates `next`): a link is a plain, orderless, non-gating prose cross-reference. Human-facing synonym: "mention" / "Mentioned in". |
| **backlink** | The reverse view of a **link**: the tasks whose body links *to* a given task via `[[id]]` — "who points at me". `App.Backlinks`/`BacklinksBatch` scan the local bodies (no server, no rate limit — the pull-side twin of GitHub's "mentioned in" panel), and `show --backlinks` renders them as a "Mentioned in" section (a `mentioned_by` array under `--json`). A task's own body mentioning itself is not a backlink, and an orphan body (a body file with no task) never surfaces since the result is drawn from the index. |
| **terminal lane** | A lane whose tasks are *not* actionable for `furrow next` — work that is finished or parked. The set is `[lanes].terminal` (default: `done`, `icebox`, `waiting`). The done lane is terminal *and* stamps `closed`; parked lanes like `icebox` and `waiting` are terminal but do **not** stamp `closed`. |
| **actionable** | The base property `core.Index.Actionable` computes: a task in a **non-terminal lane** with **all of its deps in the done lane**. `furrow next` shows the actionable tasks whose status is *also* in `[next].lanes` (default `ready`, `in-progress`) — so intake/planning lanes don't clutter "the work ready to pick up." Set `[next].lanes` to all non-terminal lanes to widen it. |
| **typed query** (`-q`) | A GitHub-Projects-style filter folded into one string, on `furrow ls -q "<query>"`. A **flat AND-list**: whitespace = AND, a comma in one value = OR, a leading `-` = NOT; `field:value` qualifiers, `has:`/`no:` presence, ordinal comparisons/ranges (`value:>=4`, `effort:2..4`), and furrow's own `is:` computed flags (`is:actionable`/`blocked`/`stuck`/`open`/`closed`/`draft`/`container`). It ANDs with the other filters, never widening a scoped board. Deliberately NO cross-field OR, grouping, or in-query sort (GitHub's own ceiling — `--json \| jq` owns the tail). Parsed by the pure `internal/query` package; `internal/app`'s `compileQuery` binds it to a task predicate. |
| **revisit** | The read-only re-evaluation query — the agent-facing counterpart to **actionable**/`next`. `core.RevisitReasons` surfaces open (non-terminal) tasks whose metadata may be out of date, each with one or more **revisit signal** `{code, detail}` pairs: `no_repo` (a draft awaiting a repo), `value_unset`, `effort_unset`, `stale`, `dep_done` (one per dep already in the done lane). `furrow revisit` renders them and attaches the signals in `--json` so an agent fixes them with the setters (`repo`/`value`/`effort`/`dep`); it never mutates. Drafts surface regardless of the board scope. An empty result is healthy and exits 0 (unlike `next`). |
| **stale** | A **revisit signal**: an open task whose `now - updated >= [revisit].stale_days` (default 30). The one signal that keys on elapsed time rather than task content; `stale_days = 0` disables it. |
| **archive** | Cold storage at `.furrow/archive/` for aged done tasks, keeping the hot store small. It is a sibling sharded store (its own `tasks/`, `meta.json`, `bodies/`), not an `archive/index.json`. `furrow archive` moves done-lane tasks whose `closed` is older than `[archive].older_than_days` (default 30; override with `--older-than`). It **previews by default** and only moves when you pass `--yes` (the destructive-op guard). |
| **the store** | The `.furrow/` directory as a whole: `tasks/<id>.json` (per-task metadata shards), `meta.json` (the layout version), `bodies/<id>.md` (prose), `bodies/assets/` (media attached via `furrow attach`, one `<id>-<name>` file per **asset**), `config.toml` (human config, read-only from furrow), and `archive/` (aged done tasks). It lives in a git repo — created by `furrow init`, committed alongside your code (or as its own central tracker repo), designed for clean git diffs. |
| **central board** | A single `.furrow` that backs **many** repos instead of each repo owning its own store — e.g. a private cross-repo tracker. Configured **user-level** via `[[board]]` entries in `${XDG_CONFIG_HOME:-~/.config}/furrow/config.toml` (covering whole trees of repos with zero per-repo files), or **per-repo** via a `.furrow-pointer.toml`. Activating a board injects a scope **repo** (usually `repo = "auto"` = the owner/repo derived from the checkout's git origin URL), which is how each task gets attached to its repo; a board's `label` is only a literal add-time tag. It is the **last** arm of discovery: a real local `.furrow` and a pointer both win over it. |
| **scope** | The directory subtree under which a user-level central board activates. A `[[board]]` lists one or more `scopes` (at least one required); the board is in effect only when cwd is under one of them. When several boards' scopes enclose cwd, the **longest (most specific) one wins**, ties broken by file order. Outside every scope, furrow behaves as if no board were configured. (`FURROW_BOARD` has no explicit scope — it derives one from the board repo's parent.) |
| **pointer** | A `.furrow-pointer.toml` at a repo root that redirects furrow to a central board (`board = <path>`) and optionally scopes the repo to one `default_repo` (a literal owner/repo, or `"auto"` to derive it). The per-repo counterpart to the user-level `[[board]]` config: it **wins over** the user-level boards but **loses to** a real local `.furrow`. |
| **doctor** | The machine-wide board-setup **health check** (`furrow doctor [dir...]`) — `boards`' opinionated sibling: `boards` lists what is configured, `doctor` says what is *wrong* with it and how to fix it. Read-only and **network-free** (it never fetches). It checks the user config parses and a usable `[[board]]` exists (`no-boards` — the half-set-up machine whose every use is a bare exit 2 naming nothing), each board's on-disk existence/readability/schema state, each scope's existence, a git-backed board's freshness vs its upstream *as of the last fetch* (`board-behind`/`board-ahead` → `furrow sync`; a mid rebase/merge warns), and where a nearer `.furrow`/pointer **shadows** the board inside its own scopes (severity `info` — a fact to see, never unhealthy). Discovery is simulated at cwd (informational) and at each argument dir (an **assertion**: `dir-unresolved` is an error carrying the fix). Findings carry stable kebab-case `code`s; exit `0` = healthy (info included), `1` = problems found (id `doctor-unhealthy`) — the health-check convention, sharable with shell init / CI. |
| **auto-filter** | The per-board `auto_filter` switch (default `true`) declaring whether the read commands (`ls`/`next`/`revisit`) scope to the board **repo**. It separates the scope repo's two jobs: the repo is what `add` attaches, `auto_filter` is whether reads are filtered by it — so `auto_filter = false` shows the whole board while writes are still attached. A **pointer** has no such knob (it always filters). Because the choice is explicit in config, filtering is **silent** (no scope banner). |
| **autocommit** | The per-board `autocommit` switch (default `false`, in the **user** config's `[[board]]`, *not* the board's committed `config.toml`) that makes furrow git-commit the board's `.furrow/` after every **mutating** command — the standalone-board convention "touch furrow → always commit" turned into a tool guarantee, so backup/undo no longer depends on remembering `furrow sync`. It **reuses `sync`'s partition rule** (machine-written files always; a co-located operator's untouched body never) plus the body *this* command wrote. It is **best-effort** (a commit failure — non-git board, clean tree, rebase in progress, `index.lock` race, gpg prompt, conflict-marker body — is a stderr warning, never a non-zero exit), it **never fetches or pushes** (that is `sync`), and an **ownership guard** skips a board whose enclosing git repo isn't its own. Per-machine, never propagated to clones/CI — contrast the presentation-only **standalone** flag. Mechanism in [`architecture.md`](architecture.md). |
| **sync** | The multi-machine board ritual as one command: auto-commit **scoped to `.furrow/`** — machine-written files (`tasks/`, `meta.json`, `config.toml`) and brand-new (untracked) bodies always commit, while a merely-modified `bodies/<id>.md` commits only when opted in via `-b/--body <id>` or `--all-bodies`, otherwise left uncommitted and reported in `pending_bodies` (plus a stderr nudge), so a shared checkout never commits a co-located operator's in-progress prose under the wrong author — → `fetch` + `rebase --autostash @{u}` → `push` (one pull→push retry on non-fast-forward). A **thin git wrapper** (`internal/gitrepo`) — git is the transport; there is no daemon and no server (see [`non-goals.md`](non-goals.md)). Per-task shards make anything but a same-task edit a clean union; a true conflict aborts the rebase and exits 3 (`sync-conflict`). The `{committed, pulled, pushed, conflict, complete, committed_bodies, pending_bodies, pending_stash}` progress object (lists omitted when empty) goes to stdout even on failure, and **`complete`** — not `pushed` — is the "fully published" flag: `false` whenever a body or stash is left pending. A successful sync also emits a repo-scoped `revisit` summary (`dep_done`/`stale` id lists), omitted when empty. The full failure-mode catalog (`sync-busy`, `sync-stash-stranded`, `sync-interrupted`, `sync-unmerged`, `body-conflict-marker`, and the silent-autostash hazard) is documented in [`architecture.md`](architecture.md). |

## The determinism contract (why "never hand-edit a shard")

`core.MarshalTask` is the single path that serializes a shard, and the bytes it
writes are meant to equal the bytes a human or an agent would type by hand
(struct-field key order, 2-space indent, `SetEscapeHTML(false)`, `[]` not `null`,
sorted + deduped label/dep sets, UTC whole-second timestamps, trailing newline).
Because of this, `Save` rewrites only the shards whose bytes changed, so an
untouched store is **zero git churn** — and a stray manual edit is exactly what
breaks it. Worse since **unknown-key passthrough** landed: a misspelled key in a
hand-edited shard (`"lables"`) is now **permanent** — furrow parks it and
faithfully writes it back forever (`furrow lint` warns `unknown-shard-key`;
removing it is a hand-edit of your own). Edit metadata through commands; edit
prose in the `bodies/*.md` files. The full byte recipe and the guards that freeze
it live in [`architecture.md`](architecture.md).

## Don't call it

Non-canonical synonyms, mapped to the term to use. Prefer the right word.

| Don't say | Say | Why |
|---|---|---|
| ticket, issue, card, todo, item, entry | **task** | One name for the unit of work. ("Issue" is especially misleading — furrow is an *alternative* to GitHub Issues, not a client of them; there is deliberately no Issues integration.) |
| database, db, store file, `index.json`, `tasks.json` | **the index** (the in-memory aggregate) / a **shard** (`.furrow/tasks/<id>.json`) | It's a set of deterministic JSON files, not a database, and not one monolithic file; each task is its own shard. "The index" is the aggregate folded from them; "the store" is the whole `.furrow/` directory. |
| description, notes, content, details, the markdown | **body** (`bodies/<id>.md`) | The prose is a separate file addressed by a relative path, not a field on the task. |
| column, board/project column, state, bucket, list, category | **lane** (the `status` field) | A lane is a `config.toml`-defined status. Two words for one thing is exactly what the glossary prevents (GitHub Projects' "column" included). |
| rank, order, weight, position, sort key | **priority** | The orderable field is `priority` — a sparse integer, not an array position. |
| slug, key, name, number, uuid, hash | **id** | The frozen handle is the `id` (e.g. `t-0042`). It is not derived from the title and never changes. |
| tag, category, topic | **label** (the `labels` field) | furrow calls them labels (mirroring GitHub Projects "Labels"). A label is a *pure* tag — the repository a task belongs to is **not** a label. |
| repository, project (as task metadata), repo label | **repo** (the `repos` field) | The `owner/repo` identifier(s) a task relates to — a first-class field, not a label convention. "Project" is GitHub's word; furrow's per-task attachment is `repos` and the whole tracker is the **board/store**. |
| draft issue, repo-less task, unattached task | **draft** (`repos: []`) | The issue-draft analogue has one name: a task with an empty `repos` set. |
| subtask, todo, checkbox, AC, acceptance criterion | **checklist item** | An in-task `checklist` entry; a *subtask* would be a separate task with a `parent`. |
| citation, source, external link | **ref** | The `refs` field — a `file:line` or URL pointer *out* of furrow. A task→task `[[id]]` reference is **not** a ref — that is a **link** (next row). |
| attachment, upload, screenshot, media file | **asset** | A file attached with `furrow attach`, stored at `.furrow/bodies/assets/<id>-<name>` and referenced from the body by a relative markdown line — never an entry in `refs`. |
| mention, wikilink, see-also, cross-reference, "mentioned in" | **link** (a `[[id]]` in a body) / **backlink** (its reverse "who points at me" view) | A task→task reference written as `[[id]]` in body prose is a **link**; the reverse view that `show --backlinks` renders is a **backlink**. "Mention" / "Mentioned in" is the human-facing synonym for the same pair. Note `@mention` (a *person*-directed notation) is **not** this — that one is unbuilt and on the roadmap (see [`non-goals.md`](non-goals.md)), not shipped. |
| blocker, requirement, prerequisite, "depends on" (as a noun) | **dep** | The `deps` field; the ids a task waits on for `next`. |
| epic | **type = epic** | An epic is a first-class `type` (a container), declared with `--type epic`; its children hang off it via `parent`. (Before v5 an epic was only a `parent` convention — that is the decision v5 reverses.) |
| group, sub-project | **parent** | The `parent` field is a single id for hierarchy, nothing heavier. |
| milestone | **(deferred)** | A time-box is a separate axis from the scope-box an epic is; furrow has no first-class milestone yet (label it, e.g. `m6`). Not a `parent` and not a `type` value. |
| closed lane, final lane, done-only | **terminal lane** | "Terminal" covers both done *and* parked (e.g. icebox); only the done lane stamps `closed`. |
| ready, available, unblocked, "the next task" | **actionable** | The precise property (non-terminal + all deps done) that `furrow next` selects on. |
| delete, purge, close out, trash | **archive** | `archive` *moves* aged done tasks to `.furrow/archive/`; it doesn't delete them, and it previews unless `--yes`. |
| migration, auto-upgrade, "furrow will convert it" | **flag day** (`furrow upgrade`) | Raising a board's layout is a deliberate, coordinated, one-way cutover a human orders — never something a write does for you in the background. Calling it a "migration" is what makes people expect it to just happen. |
| `.furrow` repo, the database, the project file | **the store** | The `.furrow/` directory; "repo" is your git repository, which contains the store. |

## What's built

Every term above is backed by shipped code (`internal/core`, `internal/app`,
`internal/cli`, `internal/gitrepo`) — furrow is **CLI-only**; the sole
presentation layer is `internal/cli`. Any interactive TUI/GUI is a separate
front-end repo that drives furrow through its CLI/JSON contract (planned:
**ridge**, a charm-v2 TUI). Shipped here: `furrow sync`, drafts/`-r` scoping,
and `furrow migrate` (importing a legacy `Task.md`). `[ui].theme` in
`config.toml` is a front-end display preference (`auto`/`dark`/`light`, with
`NO_COLOR` always honoured).
